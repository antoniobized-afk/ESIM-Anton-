# Security Gotchas

> [Назад к оглавлению](./README.md)

- Исторический `ESIMACCESS_INTEGRATION.md` содержал чувствительные доступы внутри репозитория. Это следует считать инцидентом конфигурационной безопасности; актуальный очищенный документ теперь лежит в [../../integrations/esim-access.md](../../integrations/esim-access.md).
- Admin frontend сейчас защищён browser-side PIN-кодом, а не полноценной серверной авторизацией.
- Часть admin write endpoints на backend не закрыта `JwtAdminGuard`;
- После реализации Phase 3 bot-only backend mutations нельзя вызывать из client/browser-кода: `POST /users/find-or-create` и `POST /referrals/register` теперь требуют `x-telegram-bot-token`, а user-facing профиль должен читаться через `/auth/me` или owner-guarded routes с user JWT.
- Admin JWT теперь живёт 8 часов и требует `type: 'admin'`; старые токены без `type` после деплоя будут отвергаться `JwtAdminGuard`, поэтому admin runtime должен уметь переживать принудительный re-login.
- Client redirect-параметры `returnTo` нельзя передавать напрямую в `router.push()` или `router.replace()`: все user-controlled значения должны проходить через `sanitizeRedirect()` из `client/lib/security.ts`. Sanitizer разрешает только same-origin relative URL, но сохраняет query/hash, потому что product/country/balance auto-buy flow использует вложенные внутренние `returnTo`.
- `client` intentionally не включает полный CSP в первом security hardening pass: активны только baseline headers и `frame-ancestors`, потому что Telegram Mini App и CloudPayments требуют отдельной runtime-проверки для `script-src`, `connect-src` и `frame-src`.
- URL hash с `tgWebAppData`/`tgWebAppVersion` в client считается только launch hint для ожидания Telegram SDK. Доверенная Telegram WebApp авторизация возможна только при наличии подписанного `window.Telegram.WebApp.initData` и backend-проверки `/auth/telegram/webapp`.
- Phase 18 OAuth link state нельзя трактовать как обычный `returnTo`: signed-like state с ошибочной подписью или истекшим сроком должен завершаться ошибкой привязки, а не падать в обычный OAuth login fallback.
- Phase 18 обычный OAuth login `state` тоже нельзя прокидывать в `/login/callback?returnTo=...` как есть. Backend нормализует его как same-origin relative path; external/protocol-relative/backslash/malformed значения должны превращаться в `/`.
- Phase 18 OAuth link-state secret должен быть настроен в production через `IDENTITY_LINK_STATE_SECRET` или `JWT_SECRET`. Development fallback разрешен только вне production; иначе link-state становится предсказуемым.
- Phase 18 email collision нельзя проверять только через `users.email`: явная email-привязка хранится в `UserIdentity(EMAIL, providerSubject)`, и OAuth login/link обязан считать такой email занятым даже при пустом `User.email` у владельца.
- Phase 18 contact email update (`PATCH /users/me/email`) тоже обязан проверять `UserIdentity(EMAIL, providerSubject)`. Иначе можно получить split-brain: `users.email` у одного `User`, а login email identity у другого.
- Phase 18 Telegram link/login обязан проверять `users.telegramId` как contact field. Explicit Telegram identity link не должен обновлять `User.telegramId`, но и не должен создавать split-brain, где `TELEGRAM` identity принадлежит одному `User`, а тот же `telegramId` как contact field — другому.
- Phase 18 identity create остается race-sensitive даже после preflight lookup: `P2002` на unique `(provider, providerSubject)` должен возвращаться как controlled provider conflict, а не raw Prisma error.
- Phase 18 `UserIdentity` target-state допускает максимум один active row на `(userId, provider)`. UI provider list не является достаточной защитой: backend, migration unique index и backfill preflight должны держать этот invariant самостоятельно.
- Phase 18 conflict audit нельзя писать внутри той же транзакции, которая затем бросает `ConflictException`: rollback уничтожит support/security след. `LOGIN_CONFLICT` должен писаться после rollback или до throw вне откатываемого success-path, только с hash/masked subject и без raw OAuth / Telegram payload.
- Phase 18 admin/support preflight не является "невидимым read-only" действием: он не переносит business rows, но обязан писать `MERGE_PREFLIGHT` audit для source и target user. Иначе support может просматривать merge candidates без durable security trail.
- Phase 18 user-facing identity-link endpoints не должны оставаться на inline TypeScript body types: global `ValidationPipe` реально валидирует только DTO classes. Исключение — Telegram Widget payload, где нужен исходный динамический набор полей для signature verification.
- Phase 18 legacy auth/bot endpoints, которые стали частью identity runtime, тоже не должны оставаться на inline body types: email OTP login, Telegram WebApp auth и bot `users/find-or-create` используют DTO. Telegram Widget остается исключением из-за динамического signed payload.
- VK OAuth backend route существует, но Phase 18 не включает VK в user-facing provider list автоматически. Google/Yandex/Email/Telegram остаются явным клиентским набором до отдельного продуктового решения.
