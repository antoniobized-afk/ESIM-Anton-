# Phase Authoring Guide

> Стандарт для создания новых фаз и детальных step journals в `docs/phases/`.
> Цель: чтобы фазы были одинаково читаемы, детальны и декомпозированы без хаоса.

## 1. Зачем нужен этот документ

Фаза должна быть:

- достаточно крупной, чтобы описывать законченный пользовательский или архитектурный срез;
- достаточно мелкой, чтобы её можно было выполнить и верифицировать по шагам;
- достаточно структурированной, чтобы новая сессия понимала не только **что сделать**, но и **как проверить**, что это действительно сделано.

Этот guide нужен, чтобы не появлялись:

- фазы “про всё сразу”;
- шаги, которые нельзя проверить;
- шаги, которые на самом деле являются отдельной фазой;
- journals без явного результата и зависимостей.

## 2. Что такое фаза

Фаза — это крупный инкремент системы, который:

- имеет понятную цель;
- дает конкретный результат для продукта или архитектуры;
- может быть разбит на 3-8 шагов;
- имеет свои пререквизиты, архитектурные решения и верификацию.

### Хорошие примеры фаз

- фундамент multi-tenancy
- бизнес-логика sync/analytics
- Telegram notification contour
- admin UI monitoring
- tenant cabinet

### Плохие примеры фаз

- “всё по бекенду”
- “OAuth + боты + отчеты + аналитика + деплой”
- “мелкая правка одного контроллера”

Если задача слишком мала, это не фаза, а обычная task/PR.
Если задача слишком широка, её надо делить на несколько фаз.

## 3. Как делить фазу на шаги

Каждый шаг должен быть:

- атомарным по смыслу;
- проверяемым отдельно;
- зависимым максимум от 1-3 предыдущих шагов;
- либо backend-oriented, либо UI-oriented, либо orchestration-oriented, но не всем сразу без необходимости.

### Рекомендуемое число шагов

- минимум: 3
- обычно: 4-6
- максимум без веской причины: 8

Если шагов больше 8, почти всегда фаза слишком широкая.

## 4. Как понять, что шаг выделен правильно

Правильный шаг отвечает на вопрос:

> “Можно ли сделать это как отдельный коммит/подэтап и отдельно проверить?”

Если да, шаг выделен нормально.
Если нет, шаг либо слишком размыт, либо слишком мелок.

### Хорошие шаги

- Tenant middleware и routing
- Dashboard layout и summary page
- OAuth + integration settings
- Logs page with filters

### Плохие шаги

- “backend stuff”
- “UI часть”
- “доделать всё”
- “рефакторинг”

## 5. Обязательная структура phase-файла

Каждый phase document должен содержать:

1. Заголовок фазы
2. `Цель`
3. `Результат`
4. `Оценка`
5. `Зависит от`
6. `Пререквизиты`
7. `Архитектурные решения`
8. `Шаги (журналы)`
9. `Верификация`
10. `Журнал`
11. `Ссылки на ранее созданные фазы и шаги, которые нужно учитывать при разработке этой фазы и ссылка назад на главный файл фазы из каждого шага.`

Это обязательный минимум.

## 6. Обязательная структура step journal

Каждый step document должен содержать:

1. Заголовок шага
2. `Цель`
3. `Что нужно сделать`
4. `Результат шага`

Опционально, если шаг уже реализован:

5. `Статус`
6. `Журнал изменений`
7. `Файлы`
8. `Тестирование / Верификация`

## 7. Как писать “Что нужно сделать”

Писать только проверяемые действия.

Плохо:
- “реализовать UI”
- “подключить всё”

Хорошо:
- создать middleware
- зарегистрировать routes
- добавить controller method
- рендерить page component
- провести OAuth callback flow

Каждый пункт должен быть привязан либо к файлу, либо к подсистеме, либо к поведению.

## 8. Как формировать зависимости

Зависимости указываются только там, где шаг действительно блокируется предыдущим.

### Нормально

- `6.4` зависит от `6.3`, если нужен layout и routes
- `6.2` зависит от `6.1`, если сначала нужен auth flow

### Плохо

- делать линейную цепочку всем шагам подряд “на всякий случай”
- указывать зависимость, если шаги можно делать параллельно

## 9. Как писать верификацию

Верификация должна быть сценарной, а не абстрактной.

Не так:
- “всё работает”
- “проверить руками”

Так:
- какой user flow выполняется;
- какой redirect ожидается;
- какой route отвечает;
- какой side effect должен появиться;
- какая tenant isolation должна сохраниться.

Если фаза затрагивает multi-tenancy, OAuth, background jobs или alerts, верификация должна явно это проговорить.

## 10. Что не должно попадать в фазу

- попутный refactor без прямой связи с целью фазы;
- cleanup unrelated legacy code;
- случайные идеи “на будущее” без решения;
- незафиксированные продуктовые развилки.

Если в фазе есть спорная ownership boundary или рискованный поворот архитектуры, сначала нужен ADR/wiki update, потом уже реализация.

## 11. Какой уровень детализации считать правильным

Фаза должна быть детальной enough, чтобы новая сессия могла начать работу без догадок, но не превращаться в построчный diff-план.

Хороший уровень:
- перечислены сущности, страницы, middleware, контроллеры, jobs;
- понятны зависимости;
- понятна проверка результата.

Плохой уровень:
- либо один абзац “сделать кабинет”;
- либо 50 микрошагов уровня “добавить кнопку”.

## 12. Recommended workflow для новой фазы

1. Сначала обновить [README.md](./README.md), если меняется roadmap.
2. Создать phase-файл с полной структурой.
3. Разбить фазу на 3-8 шагов.
4. Для каждого шага создать отдельный `step-XX-*.md`.
5. Зафиксировать архитектурные решения до начала кода.
6. После изменения ownership/model boundary обновить wiki.

## 12.1. Где хранить завершенные фазы

- [README.md](./README.md) — только активные, незавершенные или планируемые фазы.
- [COMPLETED_PHASES.md](./COMPLETED_PHASES.md) — архив завершенных фаз с коротким итогом и ссылкой на phase-файл.
- При закрытии фазы нужно убрать ее строку из `README.md` и добавить строку в `COMPLETED_PHASES.md`.
- Если отдельного phase-файла нет, не создавать битую ссылку: явно написать, что документа нет в текущем дереве.

## 13. Шаблоны

Для новых документов не нужно каждый раз собирать структуру вручную.

- Phase template: [templates/phase-template.md](./templates/phase-template.md)
- Step template: [templates/step-template.md](./templates/step-template.md)

Ожидаемый порядок работы:

1. Скопировать phase template.
2. Заполнить цель, результат, зависимости и верификацию.
3. Создать каталог `phase-N/`.
4. Для каждого шага скопировать step template.
5. Только после этого начинать код или детальный дизайн.

## 14. Короткий критерий качества

Хорошая фаза — это документ, по которому новый инженер или новая сессия может:

- быстро понять цель;
- увидеть границы работ;
- понять последовательность;
- проверить завершение;
- не лезть в код вслепую.
